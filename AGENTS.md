## 文档管理规范

项目中的 `DOCS.md` 文件定义了文档目录结构、命名规范和状态流转规则，请在涉及文档操作时严格遵守。

## Git 提交规范

**禁止在 commit message 中添加 Co-Authored-By 信息。**

## 代码质量规范

**每次提交代码必须确保编译通过，最好单元测试也能通过。**

**禁止因编译或单测问题而修改无关代码。** 应定位并修复真正的问题源头，而非绕道改动其他文件。

## 提交前文档更新规范

**在每次 git commit 之前，必须先根据本次代码变更更新以下文档文件。** 按顺序执行：

### 1. CHANGELOG.md

在文件顶部（`# 更新日志` 下方）新增本次变更的版本记录：

```markdown
## [vX.Y.Z] - YYYY-MM-DD

### 分类标题（如：新增功能 / Bug 修复 / 技术改进 / 性能优化 / 移除）
- **功能名称** - 简要说明
```

- 版本号从 package.json 读取
- 日期使用当天日期
- 只记录本次 commit 涉及的变更，不要复制上一版本的内容
- 如果是小改动（文档、配置），可以使用 `### 文档` 或 `### 配置` 分类

### 2. ROADMAP.md

- 更新 `最后更新` 日期和 `当前版本` 版本号
- 如果本次变更完成了 ROADMAP 中某个任务，将其状态更新为 `✅`
- 如果新增了后续计划任务，在对应阶段的"后续优化方向"表格中补充
- 更新"关键文件参考"表（如有新模块/新文件）

### 3. README.md

- 更新版本号 `**版本: vX.Y.Z**`
- 如果新增了功能类别，在"功能"部分添加对应模块和要点列表
- 保持简洁，每个功能模块 3-5 个要点即可

### 4. TODOLIST.md（仅当项目存在该文件时更新）

- 将原本 TODO 的条目，如果确认已完成则更新状态为 `✅`
- 如果有新的待办任务，添加到"待开发功能"对应阶段

### 注意事项

- 四个文件全部使用中文
- 只更新与本次变更相关的内容，不要大范围重写
- 如果某个文档不需要变更，可以跳过（但不要遗漏 CHANGELOG）
