**禁止在 commit message 中添加 Co-Authored-By 信息。**

## Git 分支开发规范

**绝不允许默认在主干（main/master）分支上直接开发，每个需求必须使用独立的特性分支。**

### 分支策略优先级

1. **Worktree 模式（优先推荐）**：使用 `git worktree add` 为新需求创建独立的工作目录，实现多分支并行开发互不干扰
2. **普通特性分支**：在当前仓库中创建新分支进行开发
3. **主干开发（禁止）**：不允许默认在 main/master 分支上直接修改

### 执行流程

1. **接到开发任务时**，首先通过 `git branch --show-current` 检测当前所在分支
2. **如果当前在主干分支上**，必须：
   - 主动询问用户使用 worktree 模式还是普通特性分支（也可根据项目情况推荐 worktree）
   - 待用户确认后，创建对应的特性分支（命名建议：`feature/<需求简述>` 或 `fix/<问题简述>`）
3. **如果已在特性分支上**，可直接在该分支上继续开发
4. **检测 worktree 模式**：可通过 `git worktree list` 确认当前是否处于 worktree 环境中

### Worktree 使用示例

```bash
# 在主仓库目录下为新需求创建 worktree
git worktree add ../project-feature-name feature/feature-name

# 开发完成后，在主仓库中清理
git worktree remove ../project-feature-name
```

### 严格禁止

- 禁止不询问用户就默认在主干分支上开发
- 禁止在未确认分支状态的情况下直接修改代码

## 代码质量规范

**每次提交代码必须确保编译通过，最好单元测试也能通过。**

**禁止因编译或单测问题而修改无关代码。** 应定位并修复真正的问题源头，而非绕道改动其他文件。

## 大批量任务执行规范（分而治之）

**用户确认全量执行后，禁止再次询问是否分批处理，必须采用「分而治之」策略自主完成。**

当任务涉及大量文件、模块或变更时，按以下流程执行：

1. **评估阶段**：分析任务规模，自行规划分批策略（按目录、模块、文件类型等维度拆分）
2. **向用户说明**（仅限首次）：简要说明分批方式和预计批次数，获取用户确认
3. **用户确认后 → 进入自主执行模式**：
   - 按批次依次处理，每批完成后**直接继续**下一批，无需再次确认
   - 每批处理完可简要汇报进度（如："已完成 3/10 批，继续处理..."），但**不得**将其变为确认请求
   - 全部完成后统一汇报最终结果

**严格禁止以下行为**（用户已确认全量执行的前提下）：
- 再次询问"文件太多，是否先处理一部分？"
- 每批处理完后停下来等待用户确认是否继续
- 以"进度汇报"的形式变相询问是否继续
- 因任务量大而拖延、缩减范围或降低质量

**仅在以下情况才允许停下询问**：
- 遇到真正的阻塞问题（权限不足、关键依赖缺失、API 不可用等）
- 发现用户需求与实际文件结构存在根本性矛盾，需要重新理解意图
- 某个中间步骤的输出会影响后续批次的处理策略，且无法自行判断

## 提交前文档更新规范

**在每次 git commit 之前，必须先根据本次代码变更更新以下文档文件。** 按顺序执行：

### 1. CHANGELOG.md

在文件顶部（`# 更新日志` 下方）新增本次变更的版本记录：

```markdown
## [vX.Y.Z] - YYYY-MM-DD

### 分类标题（如：新增功能 / Bug 修复 / 技术改进 / 性能优化 / 移除）
- **功能名称** - 简要说明
```

- 版本号从 package.json 读取
- 日期使用当天日期
- 只记录本次 commit 涉及的变更，不要复制上一版本的内容
- 如果是小改动（文档、配置），可以使用 `### 文档` 或 `### 配置` 分类

### 2. ROADMAP.md

- 更新 `最后更新` 日期和 `当前版本` 版本号
- 如果本次变更完成了 ROADMAP 中某个任务，将其状态更新为 `✅`
- 如果新增了后续计划任务，在对应阶段的"后续优化方向"表格中补充
- 更新"关键文件参考"表（如有新模块/新文件）

### 3. README.md

- 更新版本号 `**版本: vX.Y.Z**`
- 如果新增了功能类别，在"功能"部分添加对应模块和要点列表
- 保持简洁，每个功能模块 3-5 个要点即可

### 4. TODOLIST.md（仅当项目存在该文件时更新）

- 将原本 TODO 的条目，如果确认已完成则更新状态为 `✅`
- 如果有新的待办任务，添加到"待开发功能"对应阶段

### 注意事项

- 四个文件全部使用中文
- 只更新与本次变更相关的内容，不要大范围重写
- 如果某个文档不需要变更，可以跳过（但不要遗漏 CHANGELOG）
